# /nl2unit-test: 将自然语言描述转化为Web页面单元测试case

针对 Web 页面，将用户的自然语言描述解析为结构化的单元测试用例代码。

## 触发条件

TRIGGER when: 用户需要为Web页面、前端组件、UI交互生成单元测试；用户用自然语言描述测试场景并期望生成测试代码。
SKIP: 后端API测试、数据库测试、非Web相关的测试生成请求。

## 行为

### 1. 理解上下文
- 识别当前项目使用的前端框架（React、Vue、Angular 等）和测试框架（Jest、Vitest、Playwright、Cypress 等）。
- 读取相关组件/页面源码，理解 props、state、事件处理和渲染逻辑。
- 检查项目中已有的测试文件（`*.test.*`、`*.spec.*`、`__tests__/`），遵循项目既有的测试风格和命名规范。

### 2. 解析自然语言
- 从用户描述中提取：测试目标（组件/页面）、测试场景、预期行为、边界条件。
- 如果用户描述不够明确，主动追问：
  - 需要覆盖哪些交互场景？
  - 是否有异步数据加载？
  - 是否有特定的边界条件或错误状态需要覆盖？

### 3. 生成测试代码
- 根据识别到的测试框架，生成符合该框架规范的测试代码。
- 为每个测试场景生成清晰的 `describe`/`it` 分组结构（或对应框架的等价结构）。
- 测试覆盖应包括：
  - **渲染测试**: 组件是否正常渲染关键元素
  - **交互测试**: 用户点击、输入、表单提交等行为
  - **状态测试**: 不同 props/states 下的展示
  - **边界测试**: 空数据、异常数据、加载状态
  - **快照测试**（可选）: 当用户明确要求时

### 4. 输出格式
- 将测试代码写入合适的文件路径。
- 在回复中展示关键测试用例的摘要（用自然语言反向描述每个测试覆盖了什么）。
- 如果项目已有测试配置，确保生成的测试可以 `npm test` 或同等命令直接运行。

## 约束

- 优先使用 `@testing-library/user-event` 模拟用户行为（React/Vue Testing Library 场景）。
- 对外部 API 调用，建议使用项目已有的 mock 方案（msw、jest.mock 等）。
- 不要引入项目未安装的测试依赖，如需引入先提示用户安装。
- 生成的测试代码遵循项目 `CLAUDE.md` 中定义的代码规范。
- 不在测试用例中添加多余的注释，保持测试代码简洁。

## 示例

用户输入:
> 帮我给登录页面生成测试，包括：输入用户名密码点击登录、登录成功跳转、登录失败提示、以及表单为空时按钮不可点击

Skill 输出:
- 读取 `LoginPage.tsx` 源码
- 识别项目使用 React + Vitest + Testing Library
- 在 `__tests__/LoginPage.test.tsx` 生成对应测试代码
- 回复中摘要每个测试用例覆盖的场景
