# xmind2unit-test 改进设计

## 背景

本轮基于 `tests/xmind/理科精准学430.xmind` 对 `xmind2unit-test` 做了两次生成效果评估。

第一版生成结果已经证明它可以完成：

- XMind 解析与模块拆分
- 高优用例初筛
- 按模块输出 Playwright 测试文件
- 为低置信度场景保留 `FIXME`

第二版生成结果进一步证明它开始具备：

- 路径级前置条件识别
- 数据依赖识别
- 独立 mock scaffold 文件生成
- 模块级 `beforeEach` 结构化组织

但从 `tests/generated-xmind-v2` 的输出看，它距离“可直接运行的高保真 UI 自动化测试生成器”仍有明显距离：

- 复杂联动场景仍大多停在 `FIXME`
- 前置条件多数还是描述性注释，而非可执行步骤
- mock 文件仍然是脚手架，不是真正可用的 API stub
- 选择器生成仍主要依赖 OCR 文本兜底，尚未形成稳定的容器级选择器推导

因此，需要为 `xmind2unit-test` 制定一轮结构化改进方案。

## 本轮目标

本轮设计目标不是让 `xmind2unit-test` 一次性覆盖所有业务复杂度，而是建立清晰的改进优先级，并优先提升这三个核心能力：

1. **可执行前置条件生成**
2. **数据规则断言模板化**
3. **OCR → 交互容器树 → 稳定选择器映射**

## 非目标

本轮不做以下内容：

- 不追求对全部 1235 条叶子节点一次性生成可运行 case
- 不构建跨业务通用的完整页面建模系统
- 不在本轮引入视觉回归、OCR 增量训练或复杂 Agent 多轮规划链
- 不要求完全替代人工补充与业务专家确认

## 现状评估

### 已有进步

#### 1. 模块拆分和高优初筛有效

当前输出可以按一级模块稳定生成测试文件，例如：

- `教材选择器.spec.ts`
- `预习.spec.ts`
- `同步.spec.ts`
- `备考.spec.ts`

这说明它已经具备把 XMind 结构转成测试工程骨架的能力。

#### 2. 前置状态识别已出现雏形

第二版生成已经能识别诸如：

- 需要先进入弹窗/浮层状态
- 需要 mock 首次/非首次状态
- 需要 mock 加载失败或空状态

虽然目前大多只是注释，但说明路径语义提取方向是对的。

#### 3. 数据依赖识别和 mock scaffold 已经开始成立

第二版已经能把：

- 个人错因数据
- 地区/教材映射关系

这种场景识别为数据依赖，并生成：

- `mockWrongReasonDataAPI`
- `mockEditionRegionMappingAPI`

这比单纯输出 `FIXME` 已经更进一步。

### 当前主要缺口

#### 缺口 1：前置条件不可执行

当前生成结果经常知道“需要先打开弹窗”或“需要某状态”，但不会把它变成真正的步骤，例如：

- 先点击哪个入口
- 先切换到哪个 tab
- 先设置什么 mock 数据

因此，`beforeEach` 里常常只有注释，没有实际执行代码。

#### 缺口 2：复杂联动缺少结构化断言

对于：

- 自动切换小节信息
- 图谱高亮变化
- 学习进度同步更新
- 弹窗上滑到顶部

这种联动类场景，当前只能停在：

- “点击一下某个入口”
- “FIXME: 需要补充断言”

还没有把“联动结果”拆成一组可执行断言。

#### 缺口 3：数据驱动场景只识别出要 mock，不会生成规则断言

例如：

- “个人错因取 3 条，不足取本地错频 top3 补齐”
- “有映射关系时展示，反之不展示”

当前只会生成 mock 骨架与非常粗的文本断言，但不会自动生成：

- 列表长度断言
- 补齐来源断言
- 排序/映射条件断言

#### 缺口 4：OCR 仍停留在文本级，不是容器级

当前很多中低置信度 case 仍依赖：

- `getByText(/年级版本/)`
- `getByText(/Top1|Top2|Top3/)`

这种文字匹配方式，而没有推导出：

- 该文字属于哪个按钮
- 属于哪个弹窗 header/footer
- 属于 sidebar 还是 list item

这使得选择器稳定性不够。

## 改进优先级

## P0-1：可执行前置条件生成

### 目标

把“路径上推导出的前置状态”从注释提升为最小可执行步骤。

### 为什么优先

如果没有可执行前置：

- 很多高优用例会卡在错误页面状态
- 后续再强的选择器和 mock 也无法真正运行

### 设计方向

对路径中常见的状态节点建立模板化前置生成：

- `弹窗/浮层` → 生成打开入口步骤
- `首次/非首次` → 生成 mock 初始状态步骤
- `空状态/加载失败` → 生成 mock 响应步骤
- `tab/页面/详情页` → 生成页面切换/导航前置步骤

### 预期产物

- `beforeEach` 不再只有注释
- 对高优 case 至少能补出最小前置链路

## P0-2：数据规则断言模板化

### 目标

让数据依赖型 case 不只是 “mock + 文本存在性断言”，而是能生成一组和规则直接相关的断言模板。

### 为什么优先

这类场景在精准学 XMind 中出现频率高，而且很多高优用例都依赖数据规则。

### 设计方向

针对常见规则模式，自动补模板：

- `取 N 条` → `expect(locator).toHaveCount(N)`
- `不足补齐` → 生成“默认数据 + 补齐数据”两组 fixture
- `有映射展示 / 无映射不展示` → 生成正反两套场景
- `百分比/掌握度/命中率` → 生成数值断言 scaffold

### 预期产物

- mock 文件不只包含 `TODO API path`
- 测试中出现长度、可见性、正反条件的骨架断言

## P0-3：OCR 容器树到稳定选择器映射

### 目标

把 OCR 结果从“识别到哪些文字”升级为“识别到这些文字属于哪个交互容器”。

### 为什么优先

很多中低置信度 case 的核心问题，不是没识别到文字，而是不知道：

- 这个文字是按钮名
- 这个文字在弹窗 footer
- 这个文字属于左侧状态栏模块

### 设计方向

在截图分析产物中引入容器树：

- dialog / dialog-header / dialog-footer
- button / primary-button / secondary-button
- sidebar / main-content / list-item / tab-bar

语义映射阶段优先使用容器类型 + 文字生成选择器，而不是裸 `getByText(...)`。

### 预期产物

- 更高比例的 HIGH / MEDIUM 置信映射
- 选择器从 OCR 文本级别提升为交互容器级别

## P1：共享前置分组优化

### 目标

识别兄弟节点之间共享的上下文，把公共前置提取为 `test.describe(...).beforeEach(...)`，减少重复。

### 价值

- 输出测试结构更自然
- 更接近真实测试工程组织方式
- 为后续批量用例维护降低成本

## P1：Mock 路径推断

### 目标

从外部文档和项目现有代码中辅助推断更接近真实的 API 路径，而不是一律 `**/api/TODO**`。

### 价值

- 降低生成后手工补 API path 的成本
- 提升 mock scaffold 的可执行性

## 推荐分阶段实施

### 第一阶段（建议优先）

实现：

1. P0-1 可执行前置条件生成
2. P0-2 数据规则断言模板化

这两项最直接影响“生成出来的 case 能不能跑”。

### 第二阶段

实现：

3. P0-3 OCR 容器树稳定选择器映射

这项最直接影响“生成出来的选择器稳不稳”。

### 第三阶段

实现：

4. 共享前置分组优化
5. Mock 路径推断

这些更偏工程化增强。

## 成功标准

本轮后续实现只要满足以下任一提升，就算有效：

1. 第二版中的 LOW 置信用例数量明显下降
2. `beforeEach` 中出现真正可执行的前置步骤，而不是仅有注释
3. 数据规则类场景能生成长度/正反条件/补齐逻辑断言模板
4. 选择器从单纯 `getByText(...)` 提升到更多 `getByRole(...).locator(...)` 型结构

## 建议落地顺序

1. 先做 **P0-1 可执行前置条件生成**
2. 再做 **P0-2 数据规则断言模板化**
3. 之后再做 **P0-3 OCR 容器树 → 稳定选择器**

原因是：

- 前两项先解决“能不能跑”
- 第三项再解决“稳不稳”
